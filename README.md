# Conversation-to-Spec (Python Prototype)

`Conversation-to-Spec` converts messy client conversations into a structured software requirements draft for junior PMs and student project leads.

This is intentionally backend-only:
- Python-only
- CLI-only
- No frontend
- No database
- No fine-tuning in v1

## What It Produces
Given a `.txt`/`.md` conversation transcript, the pipeline outputs:
- `output/spec.json` (Pydantic-validated structured draft)
- `output/spec.md` (readable requirements document)

Structured sections:
- `project_summary`
- `functional_requirements`
- `non_functional_requirements`
- `constraints`
- `assumptions`
- `open_questions`
- `follow_up_questions`
- `question_decisions` (LLM triage: `needs_follow_up` / `already_asked` / `resolved`)
- `utterances`
- `run_metadata` (`execution_time_sec`, model/token metadata)

Each extracted requirement/note includes evidence utterance IDs (`U1`, `U2`, ...).

## Architecture (Applied)

Two pipeline modes are available:

1. **Hybrid flow (default)**:
1. Parse transcript into utterances (`U1...Un`)
2. LLM role inference (`requester` vs `developer`) per utterance
3. LLM candidate extraction from requester-side content
4. Classification model category assignment (`functional/non_functional/constraint/assumption/open_question`)
5. LLM final spec rendering (spec-style wording + ambiguity question generation)
6. Validate with Pydantic and save JSON/Markdown

2. **Legacy chunk extraction flow** (optional via config):
1. Chunk long conversations by model-aware token budget
2. Run chunk-level extraction prompts (JSON-only)
3. Retry invalid outputs in repair mode
4. Merge chunk outputs + optional consolidation/refinement
5. Validate and save outputs

### Hallucination and Ambiguity Control
- LLM performs the primary ambiguity triage and outputs `question_decisions`:
  - `needs_follow_up`
  - `already_asked`
  - `resolved`
- In hybrid mode, category assignment is delegated to a dedicated classification model.
- Optional rule post-processing can still be enabled, but defaults are now LLM-centric/lightweight.

## Installation

```bash
uv sync
```

`uv sync` will:
- create `.venv` automatically (if missing)
- install runtime + dev dependencies from `pyproject.toml`
- generate/update `uv.lock`

Optional legacy setup (pip):
```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

## CLI Usage

### 1) Single Conversation Run

Mock mode:
```bash
uv run python -m app.main --input samples/sample_conversation_1.txt --output output --mock
```

Configured model key:
```bash
uv run python -m app.main --input samples/sample_conversation_1.txt --output output --model phi_3_5_mini_instruct
```

Direct Hugging Face model name:
```bash
uv run python -m app.main --input samples/sample_conversation_1.txt --output output --model Qwen/Qwen2.5-0.5B-Instruct
```

### 2) Quantitative Evaluation

Evaluate all configured comparison models:
```bash
uv run python -m app.main --evaluate --dataset dataset/eval_samples.json
```

Evaluate one model:
```bash
uv run python -m app.main --evaluate --dataset dataset/eval_samples.json --model mock
```

Evaluation outputs:
- `eval_output/<model>/metrics.json`
- `eval_output/<model>/predictions/*`
- `eval_output/comparison_results.json`
- `eval_output/comparison_table.md`

## Evaluation Metrics (Implemented)

1. Requirement Extraction Precision / Recall / F1 (micro)
- normalized matching
- exact matching also reported

2. Requirement Type Classification Macro-F1
- categories: `functional`, `non_functional`, `constraint`, `assumption`, `open_question`

3. Evidence Linking
- evidence precision/recall/F1
- evidence Jaccard similarity

4. Hallucination Rate
- unsupported predicted requirements / total predicted requirements

5. Open Question Recall
- recall over gold open-question items

6. Schema Validity Rate
- JSON parse success rate
- Pydantic validation success rate
- first-pass (before repair) rates also reported

Additional reporting:
- average latency
- average prompt/completion tokens (when available)
- failure cluster counts (`json_error`, `schema_error`, `high_hallucination`, etc.)

## Runtime Progress Logs

The CLI now prints stage-by-stage progress logs by default, including:
- chunk planning and chunk index progress
- model loading/device info
- generation start/finish
- heartbeat logs during long generations (for perceived liveness)

Configure in `configs/prompts.yaml`:
- `progress_logging: true|false`
- `progress_heartbeat_sec: <int>`

Failure diagnostics:
- `output/error.log`: human-readable summary + embedded JSON diagnostics
- `output/error_details.json`: full structured debug payload including per-attempt raw model outputs

## LLM-Centric Mode

Default config now uses an LLM-centric profile (`configs/prompts.yaml`):
- `llm_centric_mode: true`
- heavy rule postprocessing disabled by default
- model output is preserved as much as possible (minimal structural cleanup only)

If you want stricter rule-driven cleanup again, set:
- `llm_centric_mode: false`
- then enable selected guards such as `enable_sanity_postprocess`, `enforce_requester_source_only`, and `enforce_llm_question_decisions`.

## Notes on Smaller Models

Some smaller models (for example `flan_t5_small`) may still fail strict JSON generation.
- The pipeline retries in repair mode.
- For short-context models, compact prompt profile is used.
- If chunk-level extraction fully fails, the pipeline attempts LLM-based utterance rescue before failing.
- If outputs are still invalid after retries, the run fails explicitly (no mock fallback).
- See `output/error.log` for failure details.

## Current Limitations

- Still prompt-sensitive for weaker models.
- Exact/normalized matching does not fully capture paraphrase semantics.
- No fine-tuned domain model yet.
- Hallucination guard is heuristic (evidence lexical overlap).
